    Pipeline

    How The Forecast Was Built

    01

    Aggregate Container Rows

    Convert container-level reefer records into one terminal-level hourly target with load, container counts, temperature summaries, stack-tier proxies, and customer mix.

    02

    Build Causal Features

    Create lagged load signals, rolling statistics, calendar features, and reefer-state proxies using past information only.

    03

    Train Recursive Models

    Train direct day-ahead models using only `t-24h` or older information so every target hour is predicted under the challenge's 24-hour-ahead rule.

    04

    Blend For Final Submission

    Combine an XGBoost day-ahead model and a Ridge day-ahead model using weights chosen on December validation only, then calibrate `pred_p90_kw` from that same holdout.

    Deployment Notes

    Frontend Today, Backend If Needed

    Current Demo

    Static package explorer

    The current website is a static presentation dashboard. It reads the prepared CSV files from the package and is enough for sharing the model, metrics, and outputs.

    Why no backend yet

    Not required for this package

    The challenge deliverable is a prediction file and documented approach. For that, a lightweight static site is enough and easier to host and review.

    Possible next step

    API for new uploads

    If reviewers need to test additional data later, the next version can add a small backend API to accept files, rerun the pipeline, and return a fresh forecast.

    Final Choice

    Why the final blend was selected

    The final selected submission is the strict day-ahead blend because it improved the clean no-leakage score while using only 24-hour-lagged-or-older information.

    Package Logic

    How raw files became forecasting inputs

    Raw container-level reefer rows were aggregated into a terminal-hour table, weather rows were converted into hourly environmental features, and both were used to build a reproducible forecasting workflow.

    LLM Usage

    Where AI helped

    LLMs were used for feature brainstorming, code generation, experiment design, and documentation support. Final modeling choices were validated with time-based testing.
